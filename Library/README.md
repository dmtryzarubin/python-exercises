# Bookstore Project

Учебный проект: книжный магазин, который закупает книги у поставщиков и продает их покупателям.

Первые этапы нужны не для того, чтобы сразу написать идеальный код, а чтобы научиться думать о данных: что хранить, где хранить и почему одна структура удобнее другой.

## Этап 1. Диаграмма и определение сущностей

### Цель

Перед написанием кода нужно понять, какие данные есть в книжном магазине и как они связаны между собой.

На этом этапе не нужно писать Python-код. Нужно нарисовать схему и объяснить ее словами.

### Задача

Представь, что магазин ведет учет книг. Он закупает книги, хранит их на складе и продает покупателям.

Ответь на вопросы:

1. Какие объекты реального мира есть в этой системе?
2. Какие данные нужно хранить о каждом объекте?
3. Какие действия происходят с этими объектами?
4. Как понять, сколько книг сейчас есть на складе?
5. Как потом узнать, сколько магазин потратил, заработал и какая получилась прибыль?

После этого нарисуй диаграмму: какие сущности есть в системе и как они связаны.

Можно использовать draw.io, dbdiagram.io или просто лист бумаги.

### С чего можно начать

Самый простой вариант - хранить все в одной таблице `books`:


| id  | title                 | author_name  | genre  | language | year | count | buy_price | sell_price |
| --- | --------------------- | ------------ | ------ | -------- | ---- | ----- | --------- | ---------- |
| 1   | Python Crash Course   | Eric Matthes | python | en       | 2019 | 10    | 1000      | 1500       |
| 2   | Python Crash Course 2 | Eric Matthes | python | en       | 2023 | 5     | 1200      | 1700       |


На первый взгляд это удобно: вся информация лежит в одном месте.

Но теперь подумай над ситуациями:

1. У автора поменялась фамилия, а в магазине уже 10000 его книг. Что придется обновлять?
2. Автор написал много книг. Нужно ли каждый раз повторять его имя в каждой строке?
3. Книгу закупали несколько раз по разной цене. Где хранить историю закупок?
4. Книгу продавали много раз. Как потом посчитать прибыль?
5. `count` показывает только текущее количество. А как узнать, почему оно стало именно таким?

Эти вопросы должны привести тебя к идее разделить данные на несколько сущностей.

### Результат этапа

Этап считается завершенным, если ты можешь объяснить:

1. Какие сущности нужны книжному магазину.
2. Какие поля есть у каждой сущности.
3. Почему не всегда удобно хранить все в одной таблице.
4. Какие связи есть между сущностями.
5. Как по твоей схеме можно будет выполнить закупку и продажу книги.

## Этап 2. Функции покупки и продажи книг

### Цель

Реализовать первые бизнес-функции магазина в файле `main.py`.

На этом этапе весь код пишется в одном файле. Базы данных, FastAPI и дополнительные библиотеки, кроме `os` и `json` пока не нужны.

Данные нужно хранить в JSON-файлах. Какие именно JSON-файлы понадобятся, зависит от сущностей, которые ты выделил на первом этапе.

### Где писать код

Создай у себя папку проекта на компьютере и инициализируй Git-репозиторий в папке проекта и создай файл `.gitignore`, чтобы не добавлять в историю лишние файлы (например, файлы данных или кэш):

```bash
git init
```

[Пример файла `.gitignore`](https://github.com/dmtryzarubin/python-exercises/blob/master/.gitignore)


После инициализации внутри папки создай файл `main.py` и туда скопируй это:

```python
import json

# --- ТВОЯ ЗАДАЧА НАПИСАТЬ ФУНКЦИИ ЗДЕСЬ ---
# Тебе нужно создать функции buy_book и sell_book так,
# чтобы код в самом низу этого файла заработал и не выдал ошибок.

# Данные по сущностям будем хранить в json'ах.


def read_json(path):
    with open(path, "r") as f:
        return json.load(f)


def write_json(path, data):
    with open(path, "w") as f:

# ------------------------------------------

# --- ЭТОТ КОД НЕ МЕНЯЙ. ОН ДОЛЖЕН ЗАРАБОТАТЬ ---
if __name__ == "__main__":
    print("Начинаем симуляцию магазина...")

    # 1. Закупаем книги на склад
    buy_book(
        book_name="Python Crash Course",
        count=10,
        buy_price=1000,
        sell_price=1111,
        author_name="author1",
        genre="python",
        language="en",
        year=2001,
    )

    # 2. Продаем пару книг
    sell_book(book_name="Python Crash Course", count=3)
    sell_book(book_name="Python Crash Course", count=2)

    # 3. Пытаемся продать больше, чем есть на складе
    try:
        sell_book(book_name="Python Crash Course", count=100)
        print("ОШИБКА: Твой код позволил продать 100 книг, хотя на складе их всего 5!")
    except ValueError:
        print("ОТЛИЧНО: Твой код правильно заблокировал продажу (выдал ValueError).")
```

Внизу файла уже есть демонстрационный код в блоке:

```python
if __name__ == "__main__":
    ...
```

Этот код менять не нужно. Нужно написать функции выше него так, чтобы демонстрация заработала без ошибок.

### Что нужно реализовать

Обязательные функции:

```python
buy_book(
    book_name: str,
    count: int,
    buy_price: float,
    sell_price: float,
    author_name: str,
    genre: str,
    language: str,

```

```python

```

Можно создавать дополнительные функции, если они помогают сделать код понятнее.

### Что должна делать `buy_book`

Функция `buy_book` отвечает за закупку книг на склад.

После вызова:

```python
buy_book(
    book_name="Python Crash Course",
    count=10,
    buy_price=1000,
    sell_price=1111,
    author_name="author1",
    genre="python",
    language="en",
    year=2001,
)
```

в системе должна появиться книга, если ее еще не было, и количество этой книги на складе должно увеличиться на `10`.

Также нужно сохранить факт закупки, чтобы потом можно было восстановить историю операций.

### Что должна делать `sell_book`

Функция `sell_book` отвечает за продажу книг со склада.

После вызовов:

```python
sell_book(book_name="Python Crash Course", count=3)
sell_book(book_name="Python Crash Course", count=2)
```

количество книги на складе должно уменьшиться сначала на `3`, потом еще на `2`.

Если книг на складе недостаточно, функция должна выбросить `ValueError`.

Например:

```python
sell_book(book_name="Python Crash Course", count=100)
```

не должна позволить продать 100 книг, если на складе их меньше.

### Как хранить данные

Используй JSON-файлы как простое хранилище данных.

Подумай сам:

1. Какие сущности из первого этапа нужно сохранить в JSON?
2. Должен ли каждый тип данных лежать в отдельном файле?
3. Как связать книгу с автором?
4. Где хранить текущее количество книг?
5. Где хранить историю закупок и продаж?

Начать можно с простого решения, даже если оно неидеальное. Главное - чтобы после запуска программы было видно, какие данные сохранились и как изменилась система после покупки и продажи.

### Проверка результата

Запуск:

```bash

```

Ожидаемое поведение:

1. Скрипт закупает 10 книг.
2. Скрипт продает 3 книги.
3. Скрипт продает еще 2 книги.
4. Скрипт пытается продать 100 книг.

6. В консоли появляется сообщение, что код правильно заблокировал продажу.
7. В JSON-файлах сохраняется состояние магазина.

### Важные правила этапа

- Не менять демонстрационный код внизу `main.py`.
- Не использовать базу данных.

- Ошибки бизнес-логики обрабатывать через `ValueError`.

## Этап 3.0. Подготовка окружения (venv через uv)

### Цель

Создать изолированное Python-окружение для проекта, чтобы зависимости не конфликтовали с глобальным Python на компьютере.

### Что нужно сделать

1. **Установить `uv`** (если еще не установлен):

```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
```

1. **Создать виртуальное окружение** в корне проекта:

```bash
uv venv
```

1. **Активировать окружение:**

```bash
source .venv/bin/activate
```

1. **Проверить, что окружение активировано:**
  - в начале строки терминала появился префикс `(.venv)`;
  - команда `python --version` использует интерпретатор из `.venv`.
2. **Ставить зависимости через `uv`**:

```bash
uv pip install pydantic
```

### Почему это важно

Начиная с этапа 3 и дальше, у нас появятся внешние библиотеки (`pydantic`, `fastapi`, `uvicorn` и т.д.).  
Если ставить их глобально, проект станет менее воспроизводимым.  
`venv` + `uv` делает окружение чистым и повторяемым для любого разработчика.

### Сохранение прогресса

Зафиксируй результаты этапа в Git:

## Этап 3.1. Типизация и Pydantic-схемы

### Цель

Навести порядок в данных: добавить строгую типизацию и валидацию с помощью библиотеки `pydantic`. 
Это ключевой шаг перед тем, как мы начнем пускать в нашу систему данные из внешнего мира (через API). 

Если раньше наши функции принимали длинный список аргументов, и мы могли случайно передать строку вместо числа, то теперь Pydantic будет выступать строгим "фейсконтролем".

### Где писать код

Продолжай работать в своей папке проекта. Тебе понадобится создать новый файл для схем:

```text
schemas.py
main.py (обновленный код из этапа 2)
```

### Что нужно реализовать

1. **Установи библиотеку:** выполни в терминале `uv pip install pydantic`.
2. **Создай схемы (модели) данных** в `schemas.py` для каждой сущности, которую мы определили на Этапе 1. Наследуй их от `BaseModel`.
3. **Добавь строгую валидацию:**
  - Цена (например, `buy_price` или `sell_price`) не должна быть меньше 0. Можно использовать `Field(ge=0)`.
  - Цена продажи (`sell_price`) должна быть строго больше цены закупки (`buy_price`). Для этого напиши `@field_validator` или `@model_validator`.
  - Год издания (`year`) должен быть в разумных пределах (например, от 1000 до текущего года).
4. **Обнови бизнес-логику:** Перепиши функции в `main.py` (например, `buy_book` и `sell_book`) так, чтобы они принимали Pydantic-объекты, а не длинный список из 8-10 отдельных аргументов.

### Проверка результата

1. Попробуй закупить книгу, у которой цена продажи меньше цены закупки. Программа должна автоматически упасть с ошибкой `ValidationError` от Pydantic.
2. Попробуй передать отрицательную цену закупки (например, `-100`) — должна быть ошибка валидации.
3. Бизнес-логика покупки и продажи должна работать так же, как и во втором этапе, но теперь с гарантированно правильными (провалидированными) данными.

Зафиксируй результаты этапа в Git:


## Этап 4. FastAPI и HTTP API

### Цель

Превратить текущую бизнес-логику в веб-сервис: те же операции магазина, но теперь через HTTP-ручки.

На этом этапе мы не меняем идею проекта, а меняем способ взаимодействия с ним: вместо запуска функций напрямую вызываем API.

### Где писать код

Продолжай работать в своей папке проекта. Раздели код на сервисную логику и HTTP-слой:

```text
app.py       # FastAPI-приложение и роуты
service.py   # бизнес-логика (переиспользование функций из прошлых этапов)
schemas.py   # Pydantic-схемы запросов/ответов
```

### Что нужно реализовать

1. **Установи зависимости:** `uv pip install fastapi "uvicorn[standard]"`.
2. **Создай приложение FastAPI** в `app.py`.
3. **Опиши и реализуй ручки**, которые мы хотим видеть в проекте:


| Метод | Ручка              | Что делает                            |
| ----- | ------------------ | ------------------------------------- |
| POST  | `/books/purchase`  | Закупка книги на склад                |
| POST  | `/books/sell`      | Продажа книги со склада               |
| GET   | `/books`           | Список книг                           |
| GET   | `/books/{book_id}` | Детали книги по ID                    |
| GET   | `/inventory`       | Текущие остатки на складе             |
| GET   | `/transactions`    | История операций (закупки/продажи)    |
| GET   | `/profit`          | Сводка по выручке, затратам и прибыли |
| GET   | `/health`          | Проверка, что сервис жив              |


1. **Свяжи API и бизнес-логику:** роуты должны вызывать функции из `service.py`, а не содержать всю логику внутри себя.
2. **Обработай ошибки через HTTP-коды:**
  - `404`, если сущность (например, книга) не найдена;
  - `400`, если операция невалидна (например, недостаточно книг на складе).

### Влияние на сущности и код

Этот этап может потребовать доработки сущностей и схем. Это нормально.

Например, может понадобиться:
- расширить поля в ответах (например, `current_stock` или агрегированные суммы для `/profit`);
- дописать сервисные функции, которых не хватало в этапе 2-3 (получение списка книг, транзакций, отчета).

  - получить финансовый результат через `/profit`.
3. Убедись, что невалидные операции возвращают корректные HTTP-ошибки.

## Этап 5. Проверка API скриптом

### Цель

Написать простой Python-скрипт для проверки работоспособности HTTP-ручек (API) без использования сторонних библиотек для тестирования (таких как `pytest` или `httpx`). 
Скрипт будет по очереди делать запросы к запущенному серверу и проверять, что ответы соответствуют ожиданиям.

### Где писать код

Продолжай работать в своей папке проекта. Создай файл `test_api.py` рядом с остальными:

```text
test_api.py
```

### Что нужно реализовать

1. **Запусти свой FastAPI сервер** (если он еще не запущен):
   ```bash
   uvicorn app:app --reload
   ```
2. **Напиши скрипт `test_api.py`**, который использует встроенную библиотеку `urllib` для отправки HTTP-запросов к `http://127.0.0.1:8000`.
3. **Реализуй проверки для основных сценариев:**
   - Запрос к `/health`: должен возвращать статус 200.
   - Закупка книги (`/books/purchase`): отправить POST-запрос с JSON и проверить успешный ответ.
   - Продажа книги (`/books/sell`): отправить POST-запрос и проверить успешную продажу.
   - Ошибка: попытка продать больше книг, чем есть на складе, должна возвращать ошибку (например, 400).

Пример начала файла `test_api.py`:
```python
import urllib.request
import json
import urllib.error

BASE_URL = "http://127.0.0.1:8000"

def test_health():
    req = urllib.request.Request(f"{BASE_URL}/health")
    try:
        with urllib.request.urlopen(req) as response:
            assert response.status == 200
            print("✅ GET /health работает")
    except Exception as e:
        print(f"❌ Ошибка в GET /health: {e}")

if __name__ == "__main__":
    print("Запуск проверок API...")
    test_health()
    # Добавь вызовы остальных проверок здесь
```

### Проверка результата

1. Убедись, что сервер `uvicorn` запущен в отдельном окне терминала.
2. В новом окне терминала запусти свой скрипт:
```bash
python test_api.py
```
3. Убедись, что скрипт отрабатывает без ошибок и выводит сообщения об успешном прохождении проверок.
